PurePlay Tournament API

Human-readable design specification — vendor proposal

Status: Draft for discussion Version: 0.1 Date: 2026-06-07 Next: OpenAPI specs

This document describes the contract a golf-simulator software (e.g. PurePlay) implements so that a tournament provider can host its own tournaments, handicaps, and leaderboards on its own server, and have them played on any compatible simulator bay. It is written generically so any provider can host and any vendor can implement, but the immediate goal is a single facility running its own events for its guests.

This is the human-readable system-level integration spec. It describes the whole ecosystem and the boundaries between actors. Formal OpenAPI specs follow as two separate documents, because the play API and hosted catalog API have different owners, hosts, directions, and authentication models.

OpenAPI split Provider Play API: PurePlay simulator → provider server, authenticated with bay/venue API key. Covers handshake, roster resolution, content, rounds, shots, decisions, state, leaderboards, standings, resume, and resync.

PurePlay Course Catalog API: provider server → PurePlay cloud, authenticated with vendor/cloud credentials. Covers hosted course catalog fetch, pagination, ETags, and incremental sync.

1. Overview

The simulator vendor adds support for one configurable base URL (the provider's server). From there the simulator speaks this protocol: it authenticates the bay, establishes the players present, lists playable content, retrieves a round specification, and then streams shot-by-shot results back to the server while rendering server-computed scores, handicaps, and leaderboards. The server is the authoritative source of truth for everything: scoring, handicaps, stroke allocation, stats, and round state.

Overall lifecycle

The API fits into a larger facility/tournament lifecycle. Only the play-time portion happens on every bay visit; the other steps are setup or provider-admin workflows.

  1. Provision simulator installation. Staff enters the provider server endpoint and bay/venue API key into PurePlay once per installation.
  2. Sync course catalog. The provider imports PurePlay course data either by fetching PurePlay's online catalog or by receiving a simulator push.
  3. Set up tournaments/content. The provider configures events using synced courses, holes, tees, pin labels, formats, schedules, eligibility, handicap rules, leaderboard rules, and optional tour/series membership.
  4. Set up players. Optional and parallel: the provider may maintain its own player registry, or the facility may rely on PurePlay built-in players resolved during play.
  5. Play or resume rounds. The bay handshakes, establishes the roster, lists playable content or opens a provider portal, creates or resumes a round, and reports shots, penalties, decisions, and hole completions.
  6. Show leaderboards. The provider returns server-rendered live leaderboard and scorecard display models for the active tournament/round.
  7. Show tours, series, or league standings. Optional: the provider can expose aggregate standings across multiple events, such as order of merit, season points, rankings, or league tables.

One-time provisioning (facility setup)

Before any of the protocol below runs, the simulator software must expose a general system configuration screen where facility personnel enter, once per installation:

  • Server endpoint — the provider's base URL (e.g. https://tournaments.myfacility.com).
  • API key — the bay/venue key issued by the provider, used to authenticate every request.

This is a vendor-side requirement, not an API endpoint: it is the configuration the software persists and then uses to authenticate at handshake. It is set up once by staff and not touched by guests — distinct from the per-visit, per-player credential screen that some providers require. Optionally the software may validate the pair by performing a handshake immediately and surfacing a "connection OK / failed" result on this screen.

Two separate screens, do not conflate System config (endpoint + API key) — staff, once, per installation.  ·  Player credentials (provider-hosted mode only) — guest, per visit, generated from the handshake form-spec.

2. Architecture principles

Six decisions shape the entire API. Everything else follows from them.

Principle 1 — Server-authoritative The server owns scoring, handicap calculation, stroke allocation, stats, and round state. The simulator reports raw observations (shots, lies, coordinates) and renders what the server computes. New formats and handicap rules ship server-side with no simulator change.
Principle 2 — Simulator is a thin terminal The server returns display models (leaderboard rows, scorecard cells, points, net/gross, "thru 7"), not just raw numbers. The simulator may compute optimistic local state for instant feedback, but server state is authoritative and overrides on the next sync.
Principle 3 — Strictly software → server during play The simulator is always the client in the bay/play protocol; the server never calls a simulator installation. This includes leaderboards (the sim polls) and corrections/tampering (the sim discovers them via the revision model). Any near-real-time channel is a connection the simulator dials out (SSE/long-poll), still client-initiated. Separately, the provider server may fetch course catalog data from a PurePlay-hosted online catalog service.
Principle 4 — Monotonic revision is the backbone Every round-state response carries a monotonically increasing revision. The sim acts, the server confirms with a new revision, the sim reconciles. This single mechanism powers tampering, resume, hazard drops, and crash recovery.
Principle 5 — Two player modes collapse downstream Whether players are vendor built-in or provider-hosted, after handshake the server returns one uniform resolved roster. The mode affects exactly one screen (an optional credential form); everything after is identical.
Principle 6 — Team formats are a server-issued state machine The simulator does not hard-code scramble, greensome, best-ball, or alternate-shot scoring rules. The server returns the current eligible players, active balls, required player decision, and next origin; the simulator renders controls and reports raw shots and decisions.

3. Cross-cutting concerns

3.1 Authentication

Each bay authenticates to the provider with an API key (issued per venue or per bay). The key is presented on the handshake and on every subsequent request (e.g. Authorization: Bearer <key>). The handshake exchanges the bay identity for a sessionId used as visit context for the rest of the visit; it is not a replacement bearer credential. Player credentials (provider-hosted mode) are a separate, second factor handled during player establishment — they never replace the bay key.

3.2 Units & coordinate frame

Declared once at handshake and applied to every payload thereafter, so vendors using different conventions interoperate.

  • Unit system — metric (default: metres, m/s) or imperial (yards, mph). Declared by the bay, echoed/agreed by the server.
  • Coordinate frame — declared explicitly: origin, axis orientation, handedness, and units. Default: per-hole local frame with the tee as origin. Sufficient for all stats and simpler than a course-global frame.
Why this matters Without a declared coordinate frame, 3D positions are meaningless across vendors and across holes. Every coordinate in the API (shot start/end, optional synced geometry, nextShotOrigin) is interpreted in the frame declared at handshake.

3.3 Revision & optimistic concurrency

The server's round state has a monotonic integer revision. The rules:

  • Every round-state / leaderboard / play response includes the current revision.
  • Every write (shot, penalty, hole-complete, etc.) carries the baseRevision the sim believes it is writing against.
  • If baseRevision ≠ server's current revision, the write is rejected with 409 stale_state and the current authoritative state is returned inline so the sim can reconcile immediately.
  • Any server-side mutation (a correction, a reset) bumps revision. The sim notices on its next poll, or on the next rejected write — whichever comes first.

This is plain optimistic concurrency control — no CRDTs, no merge logic. The server is truth; the sim rebuilds from truth.

3.4 Idempotency, sequencing & offline replay

Venue networks drop. The protocol assumes it.

  • Idempotency key — every write carries a client-generated key. Retried writes are de-duplicated server-side.
  • Shot identity — every reported shot carries a stable shotId plus idempotency key. Counting stroke order is separate from reported-shot identity, which matters for scramble/greensome candidate shots.
  • Sequence identity — every shot carries (roundId, holeNumber, playerId, ballId, strokeOrdinal, shotId) so the server can order, de-duplicate, and score regardless of arrival time. Sequence is server-authoritative; the sim reads active balls, required actions, and expected strokeOrdinal values from round state.
  • Replay rebase — buffered writes are replayed one at a time. After each accepted write, the sim rebases the next buffered write onto the revision returned by the server. If a write is rejected as stale, the sim drops the remaining buffer and reconciles from authoritative state.
  • Offline buffering — the sim queues writes while offline and replays them in order on reconnect. Combined with baseRevision, this means buffered writes against a since-corrected round are cleanly rejected (see tampering) rather than corrupting state.

4. The play loop

  1. Handshake. Using the endpoint + API key from one-time provisioning, the bay authenticates; declares units, coordinate frame, capabilities, and API version. Server returns the player mode, (if provider-hosted) a credential form-spec, and a sessionId.
  2. Establish players. In native mode, done on the sim based on the handshake. Built-in mode: the sim sends the player profiles it already has. Provider-hosted mode: the sim renders the credential form, collects input, sends it. In provider-portal mode, the portal may handle player flows and the launch token resolves to the selected roster when redeemed. Either way, round creation returns a uniform resolved roster (provider id, optional external identity, name, handicap value, ranking, avatar).
  3. List or browse content. In native mode, the sim retrieves tournaments/events playable by this roster and renders them in the PurePlay UI. In provider-portal mode, the sim opens the provider's web UI, which may show competitions, tours, standings, and player flows before launching a round.
  4. Select content. The sim picks content + which players play, or redeems a portal launch token. The server creates a roundId and returns the full round spec: resolved roster, sides/teams if applicable, hole array (hole number, par, stroke index, pin label, and tee assignments), format, and authoritative stroke allocations by player or side. The server prescribes play; the simulator owns the playable geometry.
  5. Play, hole by hole. On each new hole the sim sends hole-start, optionally carrying tee/aim/pin/green coordinates for server accumulation. On each shot it reports start/end coordinates + lies + shot data. Penalties, drops, and team-format decisions are reported as they happen. When the hole ends the sim sends a hole-complete report (full shot array + aggregate + end status).
  6. Render state & leaderboard. Throughout, the sim polls round state and the leaderboard (ETag + server-controlled interval) and renders server-computed scores. It detects corrections via revision bumps.
  7. Finish. After the last hole the sim shows an end screen: the server-rendered scorecard and final leaderboard.

4.1 UX integration modes (discussion proposal)

Because PurePlay's software is not released yet, the best UX integration path should be discussed between the provider community and PurePlay. Two compatible modes are worth considering. A vendor could implement one or both.

Native endpoints still matter as a fallback, for simpler installations, and for vendors who prefer a fully native experience.

4.2 Player handling modes

The handshake response declares which mode applies. Both converge on a single resolved roster (Principle 5).

Built-in playersProvider-hosted players

HandicapThe server owns the handicap value in both modes (calculated from prior results or however the provider determines). It is returned in the resolved roster for display and converted into per-hole strokeAllocations in the round spec.

Credential form-spec

Deliberately minimal — a constrained mini-form, not an arbitrary UI language. An array of field descriptors:

{
  "fields": [
    { "key": "memberNumber", "label": "Member number", "type": "text",     "required": true },
    { "key": "pin",          "label": "PIN",           "type": "password", "required": true }
  ]
}

Field type is limited to text | number | password. The sim renders labeled inputs and posts the key/value map back. The provider resolves it against its registry and returns the roster — the sim never sees registry internals.

5. Endpoints

The play-time surface is one specification (the Provider Play API), organized into two tag groups: Setup (handshake, simulator-push course catalog sync, content) and Play (the loop, state, resume, corrections). The online course-catalog fetch is a separate PurePlay-hosted spec (the PurePlay Course Catalog API), with its own server and auth. Paths are conceptual; exact shapes are finalized in the relevant OpenAPI spec.

Tag group: Setup

5.1 Setup & handshake

POST/v1/handshakeAuthenticate bay, negotiate, open session

Bay presents its API key and declares capabilities. Server returns the player mode, optional credential form-spec, and a session.

Request — bayId, apiVersion, unitSystem, coordinateFrame, capabilities (supported formats, team-format state machine support, shot-data fields, optional geometry support).

Response — sessionId, playerMode (builtin | provider_hosted), credentialForm (if provider-hosted), agreed unitSystem, server capabilities.

POST/v1/sessions/{sessionId}/playersNative mode: establish & resolve the roster

Native selection mode only. Built-in: send held player profiles. Provider-hosted: send collected credential values. In provider portal mode, the portal handles player flow and the launch token resolves the selected roster during round creation.

Response — resolved roster: array of { playerId, externalPlayerRef, name, handicapValue, ranking, avatarUrl }. In built-in mode, externalPlayerRef preserves the PurePlay identity used for future matching/resume.

5.2 Course catalog sync

This is primarily a catalog sync, not a geometry sync. The goal is to let the provider configure tournaments against the simulator's course library: course names, tee boxes, ratings/slopes, pin labels, hole numbers, par, and stroke index. Coordinates are optional enrichment.

Two ingestion options should be supported. A provider can use either or both:

  • Online catalog fetch — the provider server periodically fetches course catalog data from a PurePlay-hosted online catalog service. This is the easiest operational model: no staff member needs to open the simulator software and press sync.
  • Simulator push — a simulator installation can push the course catalog to the provider server. This works when no online catalog service exists, or when a facility has local/private course data.
  • Optional geometry enrichment — either ingestion option may include tee/aim/pin coordinates when PurePlay can expose them, so the provider can render richer scorecards and compute spatial stats.
  • Opportunistic accumulation — every hole-start payload may also carry coordinates; the server accumulates them keyed by (courseId, holeNumber, teeId, pinLabel). This enriches the catalog but is not required for play.

The online catalog fetch is outside the bay play loop: it is provider-server → PurePlay-cloud, not provider-server → simulator. The simulator remains the HTTP client for all play-time traffic. In OpenAPI, simulator push belongs to the Provider Play API; online catalog fetch belongs to the separate PurePlay Course Catalog API.

POST/v1/courses/catalogBulk-sync course catalog

Upload one or more courses with hole lists, tee definitions, rating/slope, tee-specific par where applicable, stroke index, and pin labels. Optional coordinates may be included for tee boxes, aim points, pins, and green polygons. Idempotent per courseId; re-syncing overwrites the known catalog for that course.

GETPurePlay catalog service: /v1/courses/catalogFetch online course catalog

Optional vendor-hosted surface, belonging to the separate PurePlay Course Catalog API spec. The provider server calls PurePlay's online catalog service on a schedule or on demand and imports the same CourseCatalog shape used by simulator push. Supports pagination, incremental sync via updatedSince or ETag, and vendor authentication separate from bay API keys.

Tag group: Play

5.3 Content & rounds

Content can be reached through either native selection mode or provider portal mode. In native mode, the simulator uses the listing endpoints below and renders its own UI. In provider portal mode, the provider webview handles browsing and uses a launch token to hand off into the same round-creation flow.

GET/v1/sessions/{sessionId}/contentList playable content for this roster

Tournaments/events the resolved roster may play. Each item carries enough to render a selection card (name, format, course, status, leaderboard availability).

POST/v1/sessions/{sessionId}/roundsSelect content → create a round

Pick content + participating players, or redeem a provider-portal launchToken. Server creates a roundId and returns the resolved roster + round spec.

Round spec contains: format; sides/teams if applicable; ordered hole array — per hole: holeNumber, par, strokeIndex, pinLabel, and tee assignments; and authoritative strokeAllocations keyed by player or side (already accounting for slope/course rating, playing handicap, team handicap formulas, and any match-play normalization). It may include cached coordinates for display, but the simulator owns playable geometry. Initial revision.

GET/v1/rounds/{roundId}/specRe-fetch the round spec

Idempotent retrieval (e.g. after a crash, before resume). Same content as creation.

5.4 Play loop

POST/v1/rounds/{roundId}/holes/{holeNumber}/startBegin a hole

Signals the sim is starting this hole. Carries baseRevision and may carry the hole's tee/aim/pin/green coordinates for server accumulation. The coordinates enrich the provider catalog; they are not needed for the simulator to play the hole.

POST/v1/rounds/{roundId}/holes/{holeNumber}/shotsReport one shot

One reported shot for one player. Carries shotId, playerId, sideId (if team play), ballId, shotGroupId (if multiple candidate shots are being compared), strokeOrdinal, baseRevision, idempotency key, and the shot record (start coord+lie, end coord+lie, shot data).

POST/v1/rounds/{roundId}/holes/{holeNumber}/penaltiesReport a penalty / drop

Unplayable, OOB, water/hazard, etc. Carries the penalty type, stroke cost, and the resulting drop point (coordinate + lie) so the server records where play resumes. The server reflects this as nextShotOrigin in round state.

POST/v1/rounds/{roundId}/holes/{holeNumber}/decisionsReport a player/team decision

Used when the server's current state asks the simulator to render a choice: e.g. choose the best tee shot in greensome, choose the selected ball in scramble, concede a hole, or accept a gimme. Carries baseRevision, idempotency key, sideId, playerId where relevant, and a typed decision payload such as { type: "choose_ball", selectedShotId }.

POST/v1/rounds/{roundId}/holes/{holeNumber}/completeComplete a hole for a player or side

Sent when the hole ends for a player, ball, or side depending on the format. Carries playerId/sideId/ballId as applicable, the full shot array (redundant with streamed shots — a cross-check), the aggregate stroke count, and endStatus: holed | conceded | gimme | picked_up | max_score | did_not_finish.

5.5 State & leaderboard

GET/v1/rounds/{roundId}/stateAuthoritative round state

The single source of truth the sim renders and reconciles against. Carries revision, round status, correction metadata, and per player/side: current hole, turn order, active balls, per-hole shot history, strokes, status, computed stats, activeTurn, any requiredAction, and nextShotOrigin = { coordinate, lie } (where to place the ball next — used for resume, drops, and corrections). Supports ETag.

GET/v1/rounds/{roundId}/leaderboardServer-rendered leaderboard

Poll-based. Returns ready-to-render rows (position, name, score/points, thru) plus a server-controlled pollIntervalSeconds hint and ETag. No client scoring logic. (Optional future: a sim-initiated SSE channel for near-real-time, still client-dialed.)

5.6 Tours, series & league standings

Leaderboards answer "who is leading this tournament right now?" Standings answer "where does this player/team sit across a tour, series, season, order of merit, or league?" Standings are optional and adjacent to play: useful on content-selection, pre-round, post-round, and attract screens, but not required to run a round.

GET/v1/sessions/{sessionId}/seriesList relevant tours / series / leagues

Returns series relevant to the resolved roster and/or facility: name, type (tour | series | league | season), current status, associated content, and whether standings are available.

GET/v1/series/{seriesId}/standingsServer-rendered aggregate standings

Returns ready-to-render rows for aggregate standings: position, player/team display name, points/ranking value, events played, movement, tie markers, and optional current-player highlighting. The server owns all ranking rules and tie-breaks.

5.7 Provider portal launch (discussion proposal)

Provider portal mode lets PurePlay expose a provider-hosted web UI inside the simulator software. The provider can show competitions, leaderboards, tours, series, leagues, player flows, and facility-specific content. When the player chooses to play, the webview asks PurePlay to start simulator play through a narrow host bridge command.

Discussion material for PurePlay This section is a proposed integration option, not a demand that PurePlay must choose it. The goal is to find the lowest-effort path that gives providers rich flexibility while keeping the simulator's play implementation focused and stable.

Host bridge command

// provider webview → PurePlay host bridge
PurePlay.startRound({
  launchToken: "short-lived-provider-token"
})

The bridge should be intentionally narrow. The webview does not pass the full round spec and does not become the source of truth. It only passes a short-lived token representing the provider-approved selection: content, players, teams, eligibility, and any portal-side choices.

POST/v1/sessions/{sessionId}/roundsRedeem portal launch token

Same endpoint as native round creation, but request carries { launchToken }. The server validates the token, creates or resumes the appropriate round, and returns the normal roundId + round spec. The token should be short-lived, single-use, scoped to the current provider/session, and safe to discard if launch fails.

5.8 Resume

Because round state lives server-side keyed by roundId (which outlives the bay session), stopping mid-round and picking up later — even on a different bay — works with minimal additions.

POST/v1/rounds/{roundId}/suspendStop mid-round

Transitions the round to suspended (e.g. time's up at the bay). Server may also auto-suspend on inactivity timeout.

GET/v1/sessions/{sessionId}/resumable-roundsFind rounds to resume

For the current resolved roster, lists rounds in in_progress / suspended state — including ones started on another bay. Returns roundId, the tournament, and a resume cursor (current hole/player). Without this, the sim cannot find the round to resume.

POST/v1/rounds/{roundId}/resumeRe-attach & claim a lease

Binds the round to the new sessionId and issues a lease. While leased, writes from any other session are rejected (conflict_lease / stale_state) — preventing two bays from playing the same round at once. Returns full authoritative state so the sim renders straight back into the loop from nextShotOrigin. Rejected with round_closed if the tournament window has closed.

Round status enum in_progress · suspended · completed · abandoned. Suspend/resume move between the first two; the tournament window gates whether resume is allowed.

5.9 Tamper / correction

Corrections originate out of band from the provider's own admin/web UI — never through the sim (Principle 3). A correction is simply a server-side state edit that bumps revision. The sim discovers it via the revision model and re-syncs. Polling, write rejection, and manual resync are the v1 baseline; a future SSE channel can speed up notification while still being opened by the simulator.

How the sim finds out

  1. Poll-detect — the sim's existing state/leaderboard poll sees revision changed → discards optimistic local state → re-fetches authoritative state → re-renders. Bounded by pollIntervalSeconds.
  2. Write-reject (immediate) — a shot lands mid-edit with a stale baseRevision → 409 stale_state with current state inline → the sim reconciles at once, no waiting for the next poll.
  3. Manual resync — the sim exposes an explicit "resync" / "sync now" UI control. If a player has requested an admin correction, they can press it immediately instead of waiting for polling or the next rejected write.

Manual resync behavior

Manual resync uses the existing state endpoint; it does not require new state semantics:

  1. Discard optimistic local UI state.
  2. Pause or clear unsent local buffered writes, depending on whether they were created before the correction.
  3. GET /v1/rounds/{roundId}/state.
  4. Re-render from authoritative state: revision, nextShotOrigin, activeTurn, requiredAction, scorecard, and correction metadata.

Future: sim-initiated critical update stream

A future version may add an optional SSE or long-poll channel opened by the simulator, for critical updates such as corrections and resets. This still respects the play-time direction rule because the simulator establishes the connection.

GET /v1/rounds/{roundId}/events

Event examples:
round.corrected
round.reset
score.adjusted
player.withdrawn
leaderboard.updated

Polling and manual resync should remain the v1 baseline. SSE is a progressive enhancement for faster notification.

Correction operations (admin-side; each bumps revision)

OperationEffect on state
Reset to teeTruncate the current hole's shots; nextShotOrigin = tee.
Reset to previous positionTruncate shots after N; nextShotOrigin = that shot's end coord + lie.
Reset holeClear the hole; back to its tee.
Reset full roundClear all holes; back to hole 1 tee.
Adjust scoreOverride an aggregate without touching the shot history.
DQ / withdraw playerPlayer status change.

Round state includes a small correction marker, for example { correctionReason, correctionMessage, correctedAt }, so the simulator can explain a sudden reset or score change instead of silently moving the ball.

One field carries all resets The reset target is communicated by nextShotOrigin = { coordinate, lie } in round state — "place the ball here, on this lie." The sim reads it; it never infers position. The same field also drives hazard-drop resumption.
Offline-buffer safety (the nasty case) The sim is offline, buffers shots; meanwhile an admin resets the round. On reconnect the buffered writes carry an old baseRevision → all rejected with stale_state → the sim drops its buffer, re-fetches, and resumes from nextShotOrigin. This is exactly why baseRevision on every write is mandatory, not optional — it is what stops stale buffered shots from corrupting a corrected round.

Tamper, resume, hazard drops, and crash recovery are all the same machinery: server-authoritative state + monotonic revision + baseRevision on writes + nextShotOrigin in state.

6. Data models

6.1 Shot & lie

The atomic unit of play. The server derives scoring and most stats from the stream of these.

Shot {
  shotId:        string         // stable reported-shot identity
  strokeOrdinal: integer        // counting-stroke candidate number within the active ball/side context
  playerId:     string
  sideId?:      string         // team/side, when team play is active
  ballId:       string         // one physical/virtual ball lineage
  shotGroupId?: string         // candidate shots hit from the same origin
  start: { coordinate: Vec3, lie: Lie }
  end:   { coordinate: Vec3, lie: Lie }
  data: {                      // some or all may be present, per capabilities
    ballSpeed:    number       // m/s (or mph if imperial)
    carry:        number       // m / yd
    roll:         number       // m / yd
    offline:      number       // signed lateral deviation from intended line
    curvature:    number       // shot bend
    peakHeight:   number       // apex
    club:         string       // selected club
    // ... additional vendor fields permitted
  }
}

Vec3 { x: number, y: number, z: number }   // in the declared coordinate frame
Lie  = "tee" | "fairway" | "rough" | "deep_rough" | "sand" | "green"
     | "fringe" | "water" | "recovery" | "out_of_bounds"

shotId identifies the reported shot event. strokeOrdinal identifies the counting stroke candidate. In a scramble tee shot, four players might all report candidate shots with strokeOrdinal = 1 and the same shotGroupId; after choose_ball, only the selected shot advances the active ball lineage. Non-selected candidates remain recorded for audit/stats but do not become counting strokes.

6.2 Course catalog & optional geometry

The required sync surface is the course catalog. Geometry is optional enrichment. The simulator owns playable geometry; the provider stores synced coordinates only for authoring, display, and stats.

CourseCatalog {
  courseId:    string
  name:        string
  provider:    string          // e.g. "PurePlay"
  holes:       HoleDefinition[]
}

HoleDefinition {
  holeNumber:  integer
  strokeIndex: integer
  pinLabels:   string[]        // "THU" | "FRI" | "SAT" | "SUN" | custom
  tees:        TeeDefinition[]
  aimPoints?:  Vec3[]          // optional dogleg / target points
  pins?:       { label: string, coordinate: Vec3 }[]
  green?:      Vec3[]          // optional boundary polygon
}

TeeDefinition {
  teeId:       string
  name:        string          // e.g. "58", "55", "Blue", "White"
  color?:      string
  par:         integer         // tee-specific; some courses vary par by tee
  rating?:     number
  slope?:      integer
  length?:     number          // m / yd, per declared units
  coordinate?: Vec3
}

6.3 Strokes & scoring

The server computes everything here (Principle 1). The sim renders the results and never does slope/course-rating math.

  • Handicap value — returned in the resolved roster, for display only.
  • strokeAllocations — per player or per side, per hole, in the round spec. Already accounts for slope, course rating, playing handicap, team handicap formulas, and match-play normalization. The sim renders dots/allowances; it does not allocate them.
  • Scores & points — gross, net, Stableford points (incl. variant scoring like enhanced birdie/eagle points), match status — all server-computed and returned as display models.
Match play vs stroke play Both are handled server-side. Match-play stroke normalization (lowest plays off scratch, others receive the difference) is just another allocation rule baked into strokeAllocations. The simulator is identical for both formats. 
StrokeAllocation {
  subjectType: "player" | "side"
  subjectId:   string
  holes:       { holeNumber: integer, strokes: integer }[]
}

Individual stroke play and best-ball can allocate strokes to players. Scramble, greensome, and other combined-handicap team formats can allocate strokes to sides. The resolved roster may still show each player's handicap value for display.

6.4 Team formats

Team formats are included from v1, but the simulator still does not implement format rules. The server exposes a small state machine that tells the simulator who may hit, which balls are active, and whether the players must make a decision.

ActiveTurn {
  sideId:          string
  phase:           "hit_shots" | "choose_ball" | "play_ball" | "alternate_shot" | "complete_hole"
  eligiblePlayers: string[]
  ballIds:         string[]
  origin:          { coordinate: Vec3, lie: Lie }
  requiredAction?: RequiredAction
}

RequiredAction =
  | { type: "hit_shots", players: string[], shotGroupId: string }
  | { type: "choose_ball", candidates: { shotId: string, playerId: string, ballId: string }[] }
  | { type: "concede_hole", sideId: string }
  | { type: "accept_gimme", playerId: string, ballId: string }

Decision {
  type:            "choose_ball" | "concede_hole" | "accept_gimme"
  sideId?:         string
  playerId?:       string
  selectedShotId?: string
}
Format familyHow the generic model handles it
ScrambleAll players on a side hit from the same origin with the same shotGroupId. Server returns choose_ball; sim shows candidate balls; selected shot becomes the next origin.
GreensomeSeveral tee shots are reported as candidates, then the server asks for choose_ball. After selection, the server controls alternate-shot eligibility through eligiblePlayers.
Best ballEach player has their own ballId. The sim reports each ball independently; the server decides which completed player score counts for the side and returns display models.
Alternate shotThe server returns one active ballId and the currently eligible player(s). The sim renders only those players as available to hit.

The only team-format logic the simulator needs is generic rendering: show eligible players, collect shots, show server-provided choices, and post the selected decision. New named formats can be added server-side if they fit this state-machine vocabulary.

6.5 Stats

Server-computed from the shot stream. The sim may compute its own for instant feedback, but the server is authoritative. Stats split by what data they need:

TierNeedsExamples
Non-spatialLie + shot count only (no geometry)Score, GIR, fairways hit, up-and-down, sand saves, putts.
SpatialGeometry (positions; "right of green" needs the green polygon)Proximity to pin, dispersion, driving distance vs hole length, left/right of green, strokes-gained.

Spatial stats are a capability gated on geometry availability. A vendor that never exposes geometry still gets the full non-spatial set. Geometry can arrive through online catalog fetch, simulator push, or opportunistic accumulation during play, so the spatial layer fills in as data becomes available.

7. Defaults & open items

Confirmed defaults

Decided
  • Server owns scoring, handicap value, stroke allocation, and stats. Simulator is a thin terminal.
  • Strictly software → server during play. Leaderboard is poll-based; corrections are pull-discovered. Course catalog fetch may be provider-server → PurePlay-cloud.
  • Manual resync is a required play UI affordance: the simulator can explicitly re-fetch authoritative round state after an admin correction.
  • Authentication uses the bay API key on every request. sessionId is visit context, not a replacement bearer credential.
  • Handicap value is server-owned in both player modes; strokeAllocations are server-computed by player or side (incl. match-play normalization and team handicap formulas).
  • Course catalog sync is required for tournament authoring and can be ingested either by provider-server fetch from PurePlay's online catalog or by simulator push. Coordinates are optional enrichment owned by PurePlay/simulator data. Spatial stats are gated on geometry availability.
  • Team formats are in v1 via a generic server-issued state machine: eligible players, balls, shot groups, and required decisions.
  • Coordinate frame: per-hole local (tee = origin), declared at handshake.
  • Units: metric default, declared at handshake; imperial supported.
  • Pin positions are labels (THU–SUN/custom), not dates.
  • Tours, series, and leagues are optional aggregate result surfaces. The server owns standings/ranking rules; the simulator renders returned rows.
  • Native selection mode and provider portal mode are both discussion proposals for PurePlay; either can launch the same server-authoritative play loop.
  • One system-level specification, with two OpenAPI outputs: Provider Play API and PurePlay Course Catalog API.
  • Resume supported: roundId outlives sessions; status enum + resumable-rounds listing + lease + resume cursor.

Open items to settle before/within the OpenAPI

Open
  • OpenAPI packaging — final filenames, ownership, publication location, and versioning strategy for the Provider Play API and PurePlay Course Catalog API specs.
  • Auth operations — API-key issuance, rotation, revocation, and whether keys are scoped per bay, venue, or installation.
  • Online catalog auth — authentication, tenant scoping, rate limits, and incremental sync semantics for provider-server access to PurePlay's hosted course catalog.
  • Capability negotiation surface — exact list of negotiable shot-data fields and formats, and how a vendor advertises partial support.
  • Lease semantics — lease TTL, renewal cadence, and force-takeover (admin moves a stuck round to a new bay).
  • Inactivity auto-suspend — server-side timeout value and whether it is configurable per provider.
  • Format state-machine vocabulary — exact enum values for phases and decision controls, plus the minimum UI controls the simulator must render.
  • Standings vocabulary — exact naming for tours vs series vs leagues, ranking value display, movement indicators, tie-break display, and whether standings can be scoped to player/team/facility.
  • Provider portal bridge — whether PurePlay wants to support embedded provider UI, what webview security model applies, and the exact host bridge command shape.
  • Mulligans / re-hits / practice swings — how the round spec declares allowances and how the sim flags non-counting shots.
  • Concession granularity — match play needs hole concession and match concession, distinct from per-shot gimme.
  • Optional SSE channel — whether to include the sim-initiated critical update stream in v1 or defer it to a later version.
